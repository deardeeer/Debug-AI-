    課程簡介與心法

    在開始之前,先弄清楚一件事:你不缺程式能力,你缺的是 Debug 的方法。

    §1你真正的問題

    你的問題不是「不會寫程式」。你真正的問題是:AI 幫你寫了一坨程式碼,跑起來有 bug,但你看不懂程式碼,所以不知道怎麼跟 AI 描述問題、也不知道哪裡該重寫。

    這是現在很多人遇到的「Vibe Coding 撞牆期」。重新規劃架構不會解決這個問題,因為:

    • 就算給你一個完美架構,AI 寫出來的程式碼還是會有 bug(這是必然)
    • 你還是看不懂、不知道怎麼描述 bug
    • 你只會從「在亂的程式碼上撞牆」變成「在乾淨的程式碼上撞牆」
    核心

    你真正需要的是 Debug 能力 跟 跟 AI 協作的能力。這兩個能力跟「會不會寫程式」是兩回事,可以在 3-5 小時內學會基本款。

    §2工程師的真實 Debug 流程

    工程師面對 bug 的真實過程,不是「天才靈光一閃就知道哪裡錯」,而是:

    心法

    假設 → 驗證 → 縮小範圍 → 假設 → 驗證 → 縮小範圍 → 抓到犯人。

    就像偵探辦案。你今天學的就是當偵探的三個工具。

    • 測驗一定要做,答對才能標記通過——這是驗收你真的學會
    • 模板和速查表存起來,以後每次問 bug 都用
    • 進度自動保存,本子會記得你讀到哪

    01 — FOUNDATION

    必懂的最小概念集

    不用上 HTML/CSS/JS 完整課程。你只需要這五件事。

    §1三個檔案的分工

    Apps Script + Sheets 網頁就是這三個檔案的合作:

    # 你的網頁 = 三個檔案合作

├── Index.html       ← 畫面長什麼樣(骨架 + 樣式 + 按鈕)
├── Code.gs          ← 後端邏輯(寫資料到 Sheets、讀日曆)
└── Google Sheet     ← 資料庫(所有記錄存在這)

    §2資料怎麼流動

    你在手機點「開始」按鈕
   ↓
Index.html 的 JavaScript 接到點擊
   ↓
呼叫 Code.gs 裡的函數(例如 startTask)
   ↓
Code.gs 把資料寫進 Google Sheet
   ↓
回傳「成功」訊息給 Index.html
   ↓
畫面更新,顯示「任務進行中」

    §3三種程式語言各管什麼

    語言在哪個檔案負責什麼比喻
    HTMLIndex.html畫面結構房子的骨架
    CSSIndex.html外觀樣式油漆、家具
    JavaScriptIndex.html + Code.gs互動邏輯電器、開關

    §4「前端」與「後端」

    • 前端:使用者看到的東西。在 Index.html 裡執行,跑在你的瀏覽器。
    • 後端:使用者看不到的處理。在 Code.gs 裡執行,跑在Google 的伺服器。

    為什麼這個重要?因為 debug 前你要先判斷「壞在前端還是後端」,工具不一樣、看訊息的地方也不一樣。

    §5跟 AI 協作的鐵則

    鐵則

    讓 AI 幫你寫小塊、每次寫完都驗證。

    ✘ 錯誤示範

    「幫我寫一個時間追蹤網頁」→ AI 噴 500 行 → 跑不起來 → 你不知從哪修起。

    ✓ 正確示範

    「先寫一個按鈕,點下去印 hello」→ 跑通 → 「改成印當前時間」→ 跑通 → 「現在寫進 Sheet」→ 跑通…

    02 — UNIT ONE

    錯誤訊息怎麼讀

    工程師最大的秘密:80% 的 bug,錯誤訊息已經告訴你答案了。

    真相

    大部分新手不看錯誤訊息直接截圖問 AI,等於警察報案不看監視器直接通靈。學會讀錯誤訊息,你就贏 50% 的人。

    §1錯誤的兩種類型

    類型 A:後端錯誤(Code.gs 出錯)

    長這樣:

    TypeError: Cannot read properties of undefined (reading 'getRange')
    at startTask (Code.gs:15:23)

    從上到下、從左到右讀:

    部分內容意思
    錯誤類型TypeError型別錯誤:你把東西當成它不是的東西用了
    錯誤描述Cannot read properties of undefined某個東西是空的,你卻想從它讀屬性
    出錯位置Code.gs:15:23第 15 行第 23 字元,在 startTask 函數裡

    翻譯成人話:你的 startTask 函數第 15 行,有一個東西是空的,但你卻對它呼叫了 .getRange()。

    類型 B:前端錯誤(HTML 裡的 JavaScript 出錯)

    這種錯誤不會出現在 Apps Script 編輯器,要去瀏覽器的開發者工具看(下單元教)。

    §2五個你絕對會遇到的錯誤

    錯誤訊息意思通常原因
    TypeError: Cannot read properties of undefined你想用一個不存在的東西變數名/Sheet 名打錯
    ReferenceError: X is not definedX 根本沒宣告變數/函數名拼錯
    SyntaxError: Unexpected token語法錯誤括號、引號、分號沒成對
    Range not found找不到儲存格儲存格範圍寫錯
    Authorization required權限問題需要授權或重新部署

    03 — UNIT TWO

    Logger 與 console
    偵探的放大鏡

    有時候程式不會報錯,但結果就是不對。這時候,你需要在程式碼裡插入「印出來看看」。

    §1為什麼需要這個

    常見的「靜默失敗」狀況:

    • 點按鈕後 Sheet 沒寫入,但也沒錯誤訊息
    • 計算出來的時間不對
    • 畫面顯示 undefined

    這時候單純看程式碼猜不出來。你需要讓程式自己告訴你資料流到哪壞了。

    §2工具一:Logger.log() 給 Code.gs 用

    function startTask() {
  Logger.log('🟢 startTask 被呼叫了');  // 確認函數有跑到

  const now = new Date();
  Logger.log('現在時間: ' + now);

  const sheet = SpreadsheetApp.getActive().getSheetByName('Tasks');
  Logger.log('sheet 物件: ' + sheet);  // 印 null 就是分頁名錯

  sheet.appendRow([now]);
  Logger.log('✅ 寫入完成');
}

    怎麼看 Logger 結果?在 Apps Script 編輯器手動跑這個函數(按上方「執行」),下方會出現「執行記錄」面板。

    §3工具二:console.log() 給 HTML 用

    <script>
function onStartButtonClick() {
  console.log('🟢 按鈕被點到了');

  google.script.run
    .withSuccessHandler(function(result) {
      console.log('✅ 後端回傳:', result);
    })
    .withFailureHandler(function(err) {
      console.log('❌ 後端錯誤:', err);
    })
    .startTask();
}
</script>
    注意

    手機上很難看 Console,debug 階段一律用電腦瀏覽器。

    Chrome / Edge 按 F12 或 Ctrl + Shift + I(Mac:Cmd + Option + I)→ 找到「Console」分頁。

    §4終極技巧:夾擠法

    不知道哪一行出錯時,在懷疑的區段每隔幾行插一個 log:

    function startTask() {
  Logger.log('1');  // 印出 ✅
  const sheet = SpreadsheetApp.getActive().getSheetByName('Tasks');
  Logger.log('2');  // 印出 ✅
  const lastRow = sheet.getLastRow();
  Logger.log('3');  // 印出 ✅
  const newId = generateId();  // 假設這裡有 bug
  Logger.log('4');  // ❌ 沒印出來!
  sheet.appendRow([newId, new Date()]);
}
    技巧

    3 有印、4 沒印 → 罪犯在第 3、4 行之間,就是 generateId() 出問題。

    04 — UNIT THREE

    寫高品質的Bug Prompt

    你給的線索越多,AI 越不用猜。這個能力,才是你跟 AI 協作的真正戰力。

    §1為什麼這個最重要

    學會看錯誤訊息、學會用 Logger 之後,接下來 90% 還是會交給 AI 修。差別在於:

    ✘ 爛 Prompt

    AI 亂猜 → 改錯 → 又出新 bug → 鬼打牆 10 輪

    ✓ 好 Prompt

    AI 一次找到問題 → 一次解決

    §2高品質 Bug Prompt 的 7 個元素

    編號元素範例
    1我在做什麼「我點擊『開始任務』按鈕」
    2預期結果「Tasks 分頁應該新增一列」
    3實際結果「Sheets 沒有變化」
    4錯誤訊息「Console 顯示 TypeError」或「無」
    5已驗證「Logger 顯示 startTask 有跑到,但 appendRow 之後沒印」
    6相關程式碼貼上相關函數(不是整份 500 行)
    7環境資訊「Apps Script + Sheets,Chrome 桌面」

    §3送你一個模板

    【狀況】 專案:_______ 環境:_______ 【我在做什麼】 一句話描述操作 【預期結果】 期待會發生什麼 【實際結果】 實際發生什麼 【錯誤訊息】 貼上錯誤訊息,沒有就寫「無」 【已驗證】 - Logger / console 印出什麼 - 我已經試過什麼 【相關程式碼】 只貼相關函數,不要整份 【我的問題】 你想問什麼

    05 — UNIT FOUR

    Debug 流程 SOP

    學完工具,給你一個「遇到 bug 時的標準流程」,照著做就對了。

    §1五步抓 Bug 法

    1
    看錯誤訊息有錯誤訊息 → 讀懂它(單元一)→ 八成解決。沒有 → 進入 Step 2。
    2
    確認壞在前端還是後端前端壞:看 Chrome Console(F12)。後端壞:看 Apps Script 執行記錄。
    3
    用 Logger / console.log 夾擠法在懷疑的區段插滿 log,找出「哪一行之後就壞了」。
    4
    形成假設「我懷疑是 ___ 出問題,因為 ___」
    5
    解決自己看得出來 → 改。看不出來 → 用模板寫 bug prompt 問 AI。
    關鍵

    解決後做一件事:理解原因。不要只問「怎麼修」,要問「為什麼會這樣」。這樣下次同類 bug 你自己就能解。

    §2三個鐵則

    鐵則一:一次只改一件事

    別一次改 5 個地方,出問題你不知道是哪個改壞的。改一個 → 測 → 改一個 → 測。

    鐵則二:能跑的版本一定要備份

    跟 AI 開始大改之前,把當前能跑的版本複製存檔。改壞了至少能回去。

    鐵則三:小步前進,每步都跑

    AI 給你 100 行你看不懂,不如要它一次寫 10 行、跑一次,你看懂了再繼續。

    C跟 AI 合作的 7 條金律

    1. 小塊寫:一次只要 AI 寫一個功能,不要整份噴出來
    2. 每步驗證:跑得起來再進下一步
    3. 備份能跑的版本:大改前一定存檔
    4. 加註解:請 AI 給每個函數加註解,你才看得懂
    5. 不懂就問:不要照貼,看不懂就問 AI 這段在做什麼
    6. 用模板回報 bug:7 元素都給齊
    7. 問「為什麼」不只問「怎麼修」:理解原因才能下次自己解
