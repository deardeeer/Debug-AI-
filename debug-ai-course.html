    00 — Preface

    課程簡介與心法

    在開始之前,先弄清楚一件事:你不缺程式能力,你缺的是 Debug 的方法。

    §1你真正的問題

    你的問題不是「不會寫程式」。你的問題是 AI 幫你寫了一坨程式碼,跑起來有 bug,但你看不懂程式碼,所以不知道怎麼跟 AI 描述問題、也不知道哪裡該重寫。

    這是現在很多人遇到的「Vibe Coding 撞牆期」。重新規劃架構不會解決這個問題,因為:

    • 就算給你一個完美架構,AI 寫出來的程式碼還是會有 bug(這是必然)
    • 你還是看不懂、不知道怎麼描述 bug
    • 你只會從「在亂的程式碼上撞牆」變成「在乾淨的程式碼上撞牆」
    核心

    你真正需要的是 Debug 能力 跟 跟 AI 協作的能力。這兩個能力跟「會不會寫程式」是兩回事,而它們可以在 3-5 小時內學會基本款。

    §2工程師的真實 Debug 流程

    工程師面對 bug 的真實過程,不是「天才靈光一閃就知道哪裡錯」,而是:

    心法

    假設 → 驗證 → 縮小範圍 → 假設 → 驗證 → 縮小範圍 → 抓到犯人。

    就像偵探辦案。你今天學的就是當偵探的三個工具。

    §3三個核心能力缺口

    缺口症狀解法
    看不懂程式碼結構AI 給你 300 行你不知道誰是誰學「讀程式碼」(不是寫)
    不會描述 bug只會說「壞了、不能用」學「結構化描述問題」
    不會用 debug 工具出錯只能截圖問 AI學「Console、錯誤訊息、Logger」

    §4怎麼讀這本筆記

    • 每天讀一章,讀完就標記完成,看著進度條增加會有動力
    • 練習題先想再看答案,答案點開才會顯示
    • 模板和速查表存起來,以後每次問 bug 都用
    • 不用一次讀完,本子會記得你讀到哪
    01 — Foundation

    必懂的最小概念集

    不用上 HTML/CSS/JS 完整課程。你只需要這五件事。

    §1三個檔案的分工

    Apps Script + Sheets 網頁就是這三個檔案的合作。看懂這個你就有 50% 的概念。

    # 你的網頁 = 三個檔案合作

├── Index.html       ← 畫面長什麼樣(骨架 + 樣式 + 按鈕)
├── Code.gs          ← 後端邏輯(寫資料到 Sheets、讀日曆)
└── Google Sheet     ← 資料庫(所有記錄存在這)

    §2資料怎麼流動

    記住這條路徑,你就懂網頁怎麼運作:

    你在手機點「開始」按鈕
   ↓
Index.html 的 JavaScript 接到點擊
   ↓
呼叫 Code.gs 裡的函數(例如 startTask)
   ↓
Code.gs 把資料寫進 Google Sheet
   ↓
回傳「成功」訊息給 Index.html
   ↓
畫面更新,顯示「任務進行中」

    §3三種程式語言各管什麼

    語言在哪個檔案負責什麼比喻
    HTMLIndex.html畫面結構房子的骨架
    CSSIndex.html外觀樣式油漆、家具
    JavaScriptIndex.html + Code.gs互動邏輯電器、開關
    關鍵

    你不需要會寫這三種語言,只需要看懂 AI 寫的程式碼,加上有註解就能理解大概在做什麼。看懂 > 會寫。

    §4「前端」與「後端」

    這兩個詞會一直出現,先記下:

    • 前端:使用者看到的東西。在 Index.html 裡執行,跑在你的瀏覽器。
    • 後端:使用者看不到的處理。在 Code.gs 裡執行,跑在Google 的伺服器。

    為什麼這個重要?因為 debug 前你要先判斷「壞在前端還是後端」,工具不一樣、看訊息的地方也不一樣。

    §5跟 AI 協作的鐵則

    鐵則

    讓 AI 幫你寫小塊、每次寫完都驗證。

    ✘ 錯誤示範

    跟 AI 說「幫我寫一個時間追蹤網頁」 → 它噴 500 行給你 → 跑不起來 → 你不知從哪修起。

    ✓ 正確示範

    「先幫我寫一個按鈕,點下去 Logger 印出『hello』就好」→ 跑通 → 「現在改成印出當前時間」→ 跑通 → 「現在把時間寫進 Sheet 的 A1」→ 跑通…

    每一步都小、每一步都能跑、每一步你都看得懂。這叫「漸進式開發」,是專業工程師也在用的方法。

    02 — Unit One

    錯誤訊息
    怎麼讀

    工程師最大的秘密:80% 的 bug,錯誤訊息已經告訴你答案了。

    真相

    大部分新手不看錯誤訊息直接截圖問 AI,等於警察報案不看監視器直接通靈。學會讀錯誤訊息,你就贏 50% 的人。

    §1Apps Script 錯誤的兩種類型

    類型 A. 後端錯誤(Code.gs 出錯)

    長這樣:

    TypeError: Cannot read properties of undefined (reading 'getRange')
    at startTask (Code.gs:15:23)
    at __GS_INTERNAL_top_function_call__.gs:1:8

    從上到下、從左到右讀:

    部分內容意思
    錯誤類型TypeError型別錯誤:你把東西當成它不是的東西用了
    錯誤描述Cannot read properties of undefined某個東西是空的,你卻想從它讀屬性
    出錯位置Code.gs:15:23第 15 行,第 23 個字元,函數叫 startTask

    翻譯成人話:你的 startTask 函數第 15 行,有一個東西是空的,但你卻對它呼叫了 .getRange()。

    通常是這種寫法:

    const sheet = SpreadsheetApp.getActive().getSheetByName('Task');  // ← 拼錯了,實際叫 'Tasks'
sheet.getRange('A1').setValue('hi');  // ← sheet 是 undefined,所以炸了

    類型 B. 前端錯誤(HTML 裡的 JavaScript 出錯)

    這種錯誤不會出現在 Apps Script 編輯器,要去瀏覽器的「開發者工具」看(下一單元會教)。

    §2五個你絕對會遇到的錯誤

    記住這五個,涵蓋 70% 情況:

    錯誤訊息中文翻譯通常原因
    TypeError: Cannot read properties of undefined你想用一個不存在的東西變數名字打錯、Sheet 名字打錯、函數還沒回傳就用
    ReferenceError: X is not definedX 這個東西根本沒宣告變數名字拼錯、函數名字拼錯
    SyntaxError: Unexpected token語法錯誤,程式根本跑不起來漏了括號、分號、引號沒成對
    Range not found / Sheet not found找不到指定的儲存格/分頁Sheet 分頁名稱拼錯、A1 寫成 A:1
    Authorization required權限問題第一次跑要授權、或權限範圍不足
    練習一
    讀懂這則錯誤訊息

    假設你跑你的網頁時,Apps Script 編輯器跳出這個錯誤:

    ReferenceError: stratTask is not defined
    at Code.gs:42:5

    請問:

    1. 這是哪一種錯誤(後端 / 前端)?
    2. 出錯在第幾行?
    3. 你覺得真正的問題是什麼?
    參考解答
    1. 後端錯誤(Code.gs 開頭就是線索)
    2. 第 42 行
    3. stratTask 是 startTask 的拼字錯誤(start 拼成 strat)。ReferenceError + is not defined 99% 就是名字拼錯。

    如果你答對了,你已經超越 50% 的 Vibe Coder,他們連錯誤訊息都不看。

    03 — Unit Two

    Logger 與 console
    —— 偵探的放大鏡

    有時候程式不會報錯,但結果就是不對。這時候,你需要在程式碼裡插入「印出來看看」。

    §1為什麼需要這個

    常見的「靜默失敗」狀況:

    • 點按鈕後 Sheet 沒寫入,但也沒錯誤訊息
    • 計算出來的時間不對
    • 畫面顯示 undefined

    這時候單純看程式碼猜不出來。你需要讓程式自己告訴你資料流到哪壞了。

    §2工具一:Logger.log() —— 給 Code.gs 用

    function startTask() {
  Logger.log('🟢 startTask 被呼叫了');  // ← 確認函數有跑到

  const now = new Date();
  Logger.log('現在時間是: ' + now);  // ← 確認時間對不對

  const sheet = SpreadsheetApp.getActive().getSheetByName('Tasks');
  Logger.log('sheet 物件: ' + sheet);  // ← 印 null 就是名字錯

  sheet.appendRow([now]);
  Logger.log('✅ 寫入完成');
}

    怎麼看 Logger 的結果?

    1. 在 Apps Script 編輯器手動跑這個函數(按上方的「執行」)
    2. 下方會出現「執行記錄」面板
    3. 看哪一行印出來、哪一行沒印,就知道程式跑到哪一步停了

    §3工具二:console.log() —— 給 HTML 用

    <script>
function onStartButtonClick() {
  console.log('🟢 按鈕被點到了');

  google.script.run
    .withSuccessHandler(function(result) {
      console.log('✅ 後端回傳:', result);  // 看後端回傳什麼
    })
    .withFailureHandler(function(err) {
      console.log('❌ 後端錯誤:', err);  // 看錯在哪
    })
    .startTask();
}
</script>

    怎麼看 Console 的結果?

    注意

    手機上很難看 Console,debug 階段一律用電腦瀏覽器。

    • 電腦 Chrome / Edge:按 F12 或 Ctrl + Shift + I(Mac 是 Cmd + Option + I)
    • 找到 「Console」 分頁
    • 你的 console.log 內容會出現在這裡
    • 紅色文字 = 錯誤訊息,這就是單元一的「前端錯誤」

    §4終極技巧:夾擠法

    不知道哪一行出錯時,在懷疑的區段每隔幾行插一個 Logger,看哪個有印、哪個沒印:

    function startTask() {
  Logger.log('1');  // 印出 ✅
  const sheet = SpreadsheetApp.getActive().getSheetByName('Tasks');
  Logger.log('2');  // 印出 ✅
  const lastRow = sheet.getLastRow();
  Logger.log('3');  // 印出 ✅
  const newId = generateId();  // 假設這裡有 bug
  Logger.log('4');  // ❌ 沒印出來!
  sheet.appendRow([newId, new Date()]);
  Logger.log('5');  // 沒印出來
}
    技巧

    3 有印、4 沒印 → 罪犯在第 3 行跟第 4 行之間,也就是 generateId() 出問題。

    這招超強,工程師天天用。

    練習二
    該在哪裡插 Log?

    你的網頁點「開始」按鈕後,Sheet 沒有寫入新的列,而且沒有任何錯誤訊息。

    請告訴自己:你會在哪幾個地方插入 Logger.log 或 console.log 來找出問題? 至少想 3 個位置。

    參考解答(6 個位置)
    1. HTML 的按鈕點擊處理函數一開頭 —— console.log('按鈕點到了') 確認點擊事件有觸發
    2. withSuccessHandler 裡 —— console.log('後端回傳:', result) 確認前端有呼叫到後端
    3. withFailureHandler 裡 —— console.log('失敗:', err) 看有沒有靜默失敗
    4. 後端 startTask() 函數開頭 —— Logger.log('startTask 被呼叫') 確認後端真的有被執行
    5. appendRow 那一行前後 —— Logger.log('準備寫入:', dataArray) 確認資料長對的樣子
    6. getSheetByName 之後 —— Logger.log('sheet:', sheet) 確認分頁名稱沒拼錯(印 null 就是名字錯)

    重點不是「猜出問題」,是「逐步驗證讓程式自己告訴你答案」。

    04 — Unit Three

    寫高品質的
    Bug Prompt

    你給的線索越多,AI 越不用猜。這個能力,才是你跟 AI 協作的真正戰力。

    §1為什麼這個最重要

    你學會看錯誤訊息、學會用 Logger 之後,接下來 90% 還是會交給 AI 修。差別在於:

    ✘ 爛 Prompt

    AI 亂猜 → 改錯 → 又出新 bug → 鬼打牆 10 輪

    ✓ 好 Prompt

    AI 一次找到問題 → 一次解決

    §2高品質 Bug Prompt 的 7 個元素

    編號元素範例
    1我在做什麼「我點擊『開始任務』按鈕」
    2預期結果「Tasks 分頁應該新增一列,F 欄寫入當前時間」
    3實際結果「Sheets 沒有任何變化」
    4錯誤訊息「Console 顯示:TypeError: ...」或「無」
    5已經驗證過什麼「Logger 顯示 startTask 有跑到,但 appendRow 之後的 log 沒印出來」
    6相關程式碼貼上相關函數(不是整份 500 行)
    7環境資訊「Apps Script + Google Sheets,Chrome 桌面測試」

    §3三個層次的範例對比

    爛 Prompt

    「按鈕壞了幫我修」

    普通 Prompt

    「我點開始按鈕沒反應,幫我看一下程式碼有什麼問題」[貼上 500 行]

    高品質範例

    狀況:Apps Script + Google Sheets 個人專案,Chrome 桌面測試。

    預期:點擊「開始任務」按鈕後,Tasks 分頁新增一列,F 欄寫入當前時間。

    實際:點按鈕後 Sheets 完全沒變化。

    已驗證:

    • Console 沒有錯誤訊息
    • HTML 端 console.log('按鈕點到了') 有印出來 → 點擊事件 OK
    • HTML 端 withSuccessHandler 裡的 log 沒印出來 → 後端沒回傳
    • Code.gs 端 Logger.log('startTask 被呼叫') 沒印出來 → 後端根本沒被呼叫

    問題:為什麼後端 startTask 沒有被呼叫到?

    👆 這種 Prompt,AI 一秒看出答案:HTML 的 function startTask() 跟後端 startTask 同名了,JavaScript 進入自己的 startTask 無限遞迴,不會呼叫到後端。改成 function onStartClick() 就好。

    §4送你一個 Bug 回報模板

    存起來,以後每次問 bug 都用這個模板,你會驚訝 AI 答得多準。

    【狀況】 專案:Apps Script + Google Sheets 個人時間追蹤 測試環境:Chrome 桌面 / 手機 Safari 【我在做什麼】 一句話描述操作 【預期結果】 期待會發生什麼 【實際結果】 實際發生什麼 【錯誤訊息】 貼上錯誤訊息,沒有就寫「無」 【已驗證】 - Logger / console 印出什麼 - 我已經試過什麼 - 我懷疑問題在哪 【相關程式碼】 只貼相關函數,不要整份 【我的問題】 你想問什麼
    練習三
    實戰演練

    假設你遇到這個狀況,請用上面的模板在心裡寫一份 bug 回報:

    你點「結束任務」按鈕,預期 Sheets 的 G 欄會寫入當前時間,並計算 H 欄(實際時間 = 結束 - 開始)。結果 G 欄有寫入時間,但 H 欄是 #VALUE!。Apps Script 編輯器顯示:

    TypeError: endTime.getTime is not a function
        at endTask (Code.gs:78:24)

    範例回報

    【狀況】Apps Script + Google Sheets 個人時間追蹤,Chrome 桌面測試。

    【我在做什麼】點擊「結束任務」按鈕。

    【預期】Tasks 分頁 G 欄寫入當前時間,H 欄計算「實際時間 = 結束時間 - 開始時間」。

    【實際】G 欄有寫入時間,但 H 欄顯示 #VALUE!。

    【錯誤訊息】TypeError: endTime.getTime is not a function,出在 Code.gs 第 78 行第 24 字元,在 endTask 函數裡。

    【已驗證】從錯誤訊息看,endTime 不是一個 Date 物件,所以呼叫 .getTime() 失敗。可能是從 Sheet 讀回來的時間變成字串了。

    【問題】請幫我看看 endTime 為什麼不是 Date 物件,該怎麼把它正確轉成 Date 才能呼叫 .getTime()?

    差別:這份回報已經幫 AI 排除一半可能性,定位到「Date 型別轉換」這個小範圍。AI 回答的精準度會差 10 倍。

    05 — Unit Four

    Debug 流程 SOP

    學完工具,最後給你一個「遇到 bug 時的標準流程」,照著做就對了。

    §1五步抓 Bug 法

    1
    看錯誤訊息 有錯誤訊息 → 讀懂它(用單元一的方法)→ 八成解決。沒錯誤訊息 → 進入 Step 2。
    2
    確認「壞在前端還是後端」 前端壞:看 Chrome Console(F12)。後端壞:看 Apps Script 執行記錄。
    3
    用 Logger / console.log 夾擠法 在懷疑的區段插滿 log,找出「哪一行之後就壞了」。
    4
    形成假設 「我懷疑是 ___ 出問題,因為 ___」
    5
    解決 自己看得出來 → 改。看不出來 → 用模板寫高品質 bug prompt 問 AI。
    關鍵

    解決後做一件事:理解原因。不要只問「怎麼修」,要問「為什麼會這樣」。這樣下次同類 bug 你自己就能解。

    §2三個鐵則

    鐵則一. 一次只改一件事

    別一次改 5 個地方,出問題你不知道是哪個改壞的。改一個 → 測 → 改一個 → 測。

    鐵則二. 能跑的版本一定要備份

    跟 AI 開始大改之前,把當前能跑的版本複製存檔(Apps Script 有版本控制可以用)。改壞了至少能回去。

    鐵則三. 小步前進,每步都跑

    AI 給你 100 行你看不懂,不如要它一次寫 10 行、跑一次,你看懂了再繼續。

    06 — Appendix

    速查表 & 模板

    把這頁加入書籤。實戰時隨時翻回來查。

    A錯誤訊息速查

    看到就懷疑
    Cannot read properties of undefined變數還沒被賦值就被使用 / Sheet 名字拼錯
    X is not defined名字拼錯(99%)
    Unexpected token括號、引號、分號沒成對
    Range not found儲存格範圍寫錯
    Authorization required權限沒給,跑 onOpen 或重新部署
